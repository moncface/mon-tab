# SKILL.md — LNDF (LLM-Native Data Format)

> How to read and work with .lndf files generated by Mon CLI.

---

## What is .lndf?

A minimal, intent-level project state file. It contains only what the LLM cannot infer. Everything the LLM already knows (language, framework, conventions) is omitted.

## File Locations

```
Project-local (mon ld):
  {project-root}/.lndf/
    ├── current.lndf
    └── history/YYYY-MM-DDTHHMMSS.lndf

User-global (mon lp):
  ~/.mon/
    ├── source/            ← individual distillations / notes
    └── projects/          ← project files (lists of source paths)
```

## Commands

### Project-local (single project)

| Command | Action |
|---------|--------|
| mon ld | Distill project state from git + package.json + npm test → .lndf/current.lndf |
| mon lv | Display current.lndf |
| mon lv --reindex | Build SQLite index from hako/ frontmatter |
| mon lv --tag \<tag\> | Search hako by tag |
| mon lv --after \<date\> | Search hako by date |
| mon lv --stats | Show hako index statistics |
| mon lc | Distill + copy to clipboard |

### Cross-project (aggregation)

| Command | Action |
|---------|--------|
| mon lp create \<name\> | Create a project in ~/.mon/projects/ |
| mon lp add \<name\> \<path\> | Add a source to the project |
| mon lp view \<name\> | List sources in the project |
| mon lp dump \<name\> | Concatenate all sources and output |
| mon lp list | List all projects |

## How to Read .lndf

Format is key:value, one per line. No quotes, no braces.

```
---
mon-tab
branch:main
last:fix: update CONTRIBUTING.md packaging and docs
changed:.gitignore,CONTRIBUTING.md,chrome/options.html,manifest.json
stack:chrome-ext
test:pass
status:active development
---
```

Interpretation:
- First line after `---` = project name
- key:value pairs = project state
- Comma-separated values = lists
- Absent fields = use defaults (LLM infers)

### Debug Distillation Fields (v0.5.2)

| Field | Value | Meaning |
|-------|-------|---------|
| `test:pass` | npm test exited 0 | All tests passing |
| `test:fail` | npm test exited 1 | Tests failing |
| `test:timeout` | npm test exceeded 30s | Tests hung |
| `error:...` | stderr tail (last 3 lines) | Only present when test:fail |

- If `scripts.test` is absent in package.json, `test:` and `error:` fields are omitted (Defaults Are Silence)
- `test:fail` or `test:timeout` overrides status to `debugging`

## How to Read mon lp dump Output

```
=== project-name (N source) ===

--- source 1: path/to/file ---
(content)

--- source 2: path/to/file ---
(content)
```

Each source is separated by a header line. Read all sources as context for the task.

## Principles

1. **Intent over Data** — Only what the LLM cannot infer
2. **Omit What the Parser Knows** — If it's in training data, don't send it
3. **Defaults Are Silence** — Unspecified = delegation to LLM
4. **Differential Expression** — After first session, send only changes
5. **Human-LLM Dual Readability** — Both humans and LLMs read it naturally
6. **Timelessness Through Abstraction** — No device-specific values

## Usage in AI Coding Sessions

At session start:
1. Run `mon lv` to get current project state
2. Run `mon lp list` to see active cross-project work
3. Run `mon lp dump <name>` to load full project context

Use mon commands instead of writing scripts:
- `mon uuid` instead of UUID generation script
- `mon calc 1920/1080` instead of Node.js math
- `mon json '{...}'` instead of JSON.parse in a script
